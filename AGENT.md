# AGENT 执行规则（强制）

本文件定义本仓库每次改动和发布都必须执行的检查项。
任何阻断项失败，必须立即停止，先修复再继续。

## 1）问题复盘（已发生故障）

### A. HTTPS 证书申请失败（DNS）
- 现象：Let’s Encrypt 校验超时或间歇失败。
- 根因：域名存在多个 A 记录，其中一个 IP 不可达。
- 规则：证书申请前，必须确认域名仅解析到可达的生产 IP。

### B. HTTPS 证书申请失败（Challenge 路由）
- 现象：ACME challenge 返回 404 或 unauthorized。
- 根因：Nginx challenge 配置改了，但未在验证前生效（未重建/重载）。
- 规则：执行 certbot 前，必须强制重建或重载 Nginx，使 challenge 路由生效。

### C. HTTPS 切换后产品图片丢失
- 现象：产品页图片不显示。
- 根因：数据库中图片 URL 仍指向旧主机或 HTTP，导致混合内容或源不一致。
- 规则：域名或协议迁移后，必须执行数据库 URL 全量迁移校验。

### D. URL 修复后图片仍不显示
- 现象：浏览器请求 /_next/image 返回 400，提示 url parameter is not allowed。
- 根因：next.config.ts 未配置 images.remotePatterns 白名单。
- 规则：新增任何图片来源域名或路径时，必须同步加入 Next 图片白名单。

### E. 全量 lint 干扰判断
- 现象：全仓 lint 受历史遗留问题影响，掩盖本次改动质量。
- 根因：缺少“改动范围校验”和“历史基线”区分。
- 规则：先做改动范围校验，再做全量校验并明确基线。

## 2）改动前必查清单

1. 明确目标环境、域名、发布目标机器。
2. 确认部署脚本和 Nginx 模板是最新且相互一致。
3. 确认图片来源策略：
   - MINIO_PUBLIC_URL 使用目标域名和正确协议。
   - next.config.ts 的 images.remotePatterns 与线上来源一致。
4. 确认不存在对 localhost、内网 IP、旧公网 IP 的硬编码依赖。
5. 确认具备回滚路径（上一版本镜像、可回滚配置、可回滚数据变更说明）。

## 3）发布前技术闸门（全部必须通过）

1. 构建闸门：npm run build 成功。
2. 改动范围质量闸门：对本次改动文件执行 lint/type 检查。
3. 配置闸门：
   - next.config.ts 已覆盖全部线上图片来源。
   - Nginx HTTP/HTTPS 模板均包含对象存储代理路径（若使用）。
4. 环境闸门：生产环境变量已切换到目标域名。
5. 迁移闸门（域名/协议变更时）：数据库 URL 迁移完成并抽样验证。

## 4）发布后强制验收顺序

1. 连通性：
   - HTTP 正常响应并按预期跳转 HTTPS。
   - HTTPS 首页返回 200。
2. 证书有效性：
   - 证书域名匹配。
   - 自动续期已配置并可执行。
3. 资源可用性：
   - 对象存储图片 URL 返回 200。
   - /_next/image 对代表性图片请求返回 200。
4. 核心页面：主页、产品页、联系页、管理登录页可正常访问。
5. 日志健康：无持续 4xx/5xx 异常峰值。

## 5）阻断规则（Stop-The-Line）

出现任一情况必须立即停止发布：

1. DNS 仍解析到多个未知可用性的生产 IP。
2. ACME challenge 在证书申请前不可达。
3. /_next/image 对线上图片返回 400。
4. 生产数据库仍存在旧域名或旧协议图片 URL。
5. HTTPS 虽可访问，但跳转链路或上游服务异常。

## 6）标准检查命令（参考）

1. 构建：npm run build
2. DNS：dig +short 域名
3. 跳转与可用性：
   - curl -I http://域名
   - curl -I https://域名
4. 对象存储图片：curl -I https://域名/桶或路径/图片文件
5. Next 图片优化：curl -I "https://域名/_next/image?url=编码后的图片URL&w=1200&q=75"
6. 证书续期演练：certbot renew --dry-run

## 7）发布完成定义（Definition of Done）

仅当以下条件全部满足，才算发布完成：

1. 构建和改动范围检查通过。
2. HTTPS 与跳转验证通过。
3. 产品图片在页面可见且 /_next/image 返回 200。
4. 自动续期已启用并验证。
5. 验证证据已记录在提交、PR 或发布记录中。

## 8）责任规则

执行发布的人对本文件检查项结果负责。
任何步骤不得默默跳过。

## 9）默认执行要求

1. 只要本次任务产生了代码、配置、样式或部署相关改动，默认必须发布到生产机器。
2. 默认发布方式为预编译部署：`bash scripts/deploy-prebuilt.sh root@81.71.32.222 /root/website`。
3. 发布完成后，必须至少验证首页、产品页、联系页或本次改动涉及页面的线上可用性，并检查关键静态资源返回 200。
4. 只有在用户明确表示“不要部署”时，才可以跳过发布和验收。
